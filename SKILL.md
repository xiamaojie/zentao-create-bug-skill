# 禅道提 Bug 规则

按以下顺序执行，优先保证 Bug 落点正确，其次保证字段完整，最后再追求格式一致。

## 输入解析

优先从用户输入中提取以下字段：

- `产品`
- `项目`
- `指派给`
- `标题`
- `现象`
- `复现步骤`
- `期望结果`
- `实际结果`
- `图片路径`
- `附件路径`
- `要求`

允许用户只提供自然语言，不要求固定模板。

若用户未提供 `标题`，根据 `现象` 自动生成一个简洁标题。

若用户同时提供 `项目` 和 `产品`，优先采用用户显式给出的 `产品`，并将 `项目` 作为可见性挂载字段。

若用户提供本地图片或附件绝对路径，例如：

- `图片路径：/Users/admin/Downloads/a.png`
- `附件：/Users/admin/Downloads/a.png, /Users/admin/Downloads/b.jpg`

则应先分别解析：

- `图片路径` -> 内嵌图片列表
- `附件路径` / `附件` -> 附件列表

## 实体解析

### 用户说“产品”时

只解析产品，不主动反查项目。

创建 Bug 时必须显式传：

- `product=<产品ID>`

仅当用户明确要求在某个项目视图可见时，再补充：

- `project=<项目ID>`

### 用户说“项目”时

先查询项目，再解析项目关联产品。

禁止直接把项目 ID 当成产品 ID 使用，除非接口数据明确表明该项目关联产品就是同一个 ID。

创建 Bug 时必须显式传：

- `project=<项目ID>`
- `product=<关联产品ID>`

## 查询策略

优先使用原始 `/projects`、`/products` 接口做名称检索。

若接口支持服务端过滤，直接使用过滤结果继续解析。

若接口接受了查询参数但返回结果未过滤，禁止据此认定“项目/产品不存在”，必须回退执行原始全量分页确认：

1. 分页读取 `/projects`
2. 分页读取 `/products`
3. 按名称精确匹配
4. 再按包含匹配作为候选补充

若通过全量分页确认到唯一落点，允许继续创建，并在最终结果中说明：

- 本次是通过原始全量分页确认准确落点
- 最终匹配到的项目名 / 产品名
- 若用户给的是简称，也要说明系统内的真实名称

若用户给的是英文简称，允许使用名称前缀做保守模糊匹配：

1. 先将名称按空格、连字符、下划线归一化分词
2. 优先比较前两个有效单词
3. 若用户输入与系统名称前两个单词一致，且候选唯一，可直接视为命中落点

例如：

- 用户输入 `Wallpaper Spark`
- 系统名称为 `Wallpaper Spark Launcher`

在这种情况下，可将 `Wallpaper Spark Launcher` 视为命中落点。

但必须同时满足以下约束：

- 命中后候选唯一
- 不存在另一个名称前两个单词也相同的项目或产品
- 不能跨项目/产品类型随意替代，仍需分别确认项目候选和产品候选

若全量分页后仍存在多个合理候选，不要猜测，要求用户确认。

## 产品判定

### 真实产品与影子产品

查询产品后，检查：

- `shadow`
- `bind`
- 产品名称
- 项目关联关系
- 历史 Bug 落点

按以下优先级选择最终产品：

1. 用户显式指定的产品
2. 该项目历史 Bug 一致落点的产品
3. 非影子产品
4. 项目当前唯一关联产品
5. 影子产品

若最终只能挂影子产品，允许创建，但必须在结果中明确说明：

- 当前 Bug 挂载到影子产品
- 对应项目是什么
- 对应产品是什么

若同时存在多个合理候选且无法稳定判断，不要猜测，向用户说明存在歧义并要求确认。

## 指派人解析

用户通常给中文姓名。必须先查询用户列表，将中文姓名映射成禅道账号。

例如：

- `陈润国` -> `chenrunguo`

禁止直接把中文姓名写入 `assignedTo`。

若姓名无法唯一匹配：

1. 先尝试精确匹配 `realname`
2. 再尝试拼音或账号近似
3. 若仍不唯一，要求用户确认

## 字段补全

若用户未指定以下字段，按默认规则补全：

### 优先级与严重程度

使用禅道常见默认值：

- `pri=3`
- `severity=3`

### 类型

按现象推断：

- 闪退、崩溃、异常退出、报错：`codeerror`
- 界面展示异常：优先 `ui`
- 逻辑与结果错误：优先 `codeerror`

若无法判断，默认：

- `type=codeerror`

### 模块

优先使用：

1. 该产品历史 Bug 常见模块
2. 根模块 `0`

### 版本 / 构建

优先使用：

1. 项目或产品已有构建
2. 若没有可用构建，使用 `主干`

### 环境字段

若用户提供设备和系统信息，尽量写入：

- `os`
- `hardware`

若接口实际未落库这些字段，不要因此阻塞创建。

## 标题生成规则

若用户未显式给标题，按以下模板生成：

`<系统/设备> <核心现象>`

示例：

- `Android 16 Pixel 6 打开 PDF 后切后台再返回闪退`

标题要求：

- 保留核心现象
- 控制在一行内
- 避免包含冗余说明
- 不重复写“Bug”“问题”这类无效词

## 步骤整理规则

将用户提供内容整理到 `steps` 字段，推荐格式：

```text
[现象]
...

[复现步骤]
1. ...
2. ...
3. ...

[期望结果]
...

[实际结果]
...
```

若用户把多个段落写在同一行，例如：

`[现象] ... [复现步骤] ... [期望结果] ... [实际结果] ...`

则必须先做输入规范化，再提交到 `steps`：

1. 识别段落标签：`[现象]`、`[复现步骤]`、`[期望结果]`、`[实际结果]`
2. 每个标签单独起一行
3. 各段内容与标签之间保留换行
4. 段与段之间补一个空行
5. 若 `复现步骤` 中的内容仍是一整段，继续整理成编号列表

规范化后的结果应尽量接近：

```text
[现象]
...

[复现步骤]
1. ...
2. ...
3. ...

[期望结果]
...

[实际结果]
...
```

若步骤中包含 JSON、配置字典、键值对片段，默认保留多行缩进样式，不要压成单行。

推荐处理方式：

1. 将配置内容放在对应步骤的下一行
2. 保留原有键顺序
3. 保留多行缩进
4. 与步骤说明文字之间保留换行

例如：

```text
1. 配置如下在线参数：
{
  "Daily_Limit": 99,
  "Only_Show_For_Promote": 0,
  "Life_Time_Limit": 10
}
```

若原始输入中的 JSON 被压成一行，整理时应自动展开为便于阅读的多行缩进格式。

若用户未提供“现象”，可以省略该段。

若用户复现步骤不是编号列表，整理成编号列表再提交。

## 图片与附件规则

当用户提供 `图片路径`、`附件路径`、`附件` 时，按以下规则处理：

1. 只接受本地文件路径，不要把会话里的 `[Image #1]`、截图占位符、URL 当成可上传素材
2. 优先识别绝对路径
3. 多个路径时按用户给出的顺序保留
4. 创建前先检查路径非空且文件存在
5. `图片路径` 默认不是附件，而是内嵌图片；应优先放到 `steps` 末尾，效果等同于在禅道富文本编辑器里“直接贴图”
6. 若运行时工具支持 `inlineImagePaths`，必须优先使用该能力；不要把 `图片路径` 降级成 `attachmentPaths`
7. 只要本次存在 `图片路径`，就必须优先走支持表单 / 富文本处理的创建入口，例如支持 `inlineImagePaths` 的 `create_bug`；禁止退回普通 `/bugs` REST 创建入口，否则内嵌图片会丢失
8. 若当前运行时需要自行拼装描述内图片，必须先用同一个 `uid` 调用类似 `/api.php/v1/files?uid=<uid>` 的上传接口，把本地图片上传成禅道 `file-read-<id>.<ext>` 资源，再把返回 URL 写入 `steps` 的 `<img src=\"...\">`；不要直接写 `data:` base64 图片
9. 描述内图片默认应带较小的显示样式，优先使用 `style=\"max-width:360px;max-height:520px;height:auto;\"`，避免图片在详情页撑满内容区，且过长图片不会占满整屏
10. `附件路径` / `附件` 仍按附件处理；若运行时工具支持 `attachmentPaths` 或 multipart 上传，则创建 Bug 时直接一并上传
11. 若附件创建时上传成功但 `files` 为空，或优先策略失败，则必须在拿到 `bugID` 后立即执行“创建后补传附件”
12. 只有在当前运行时明确既不支持内嵌图片，也不支持附件自动上传时，才允许告诉用户需要手动处理

若用户给的是会话图片占位符而不是真实路径，必须明确说明当前只能读取图片内容，不能直接上传为禅道附件，并要求用户提供本地文件路径。

## 创建流程

按以下顺序执行：

1. 识别用户输入中的产品名或项目名
2. 查询产品或项目
3. 若输入的是项目，解析关联产品
4. 判断最终挂载的是真实产品还是影子产品
5. 查询指派人账号
6. 补全默认字段
7. 调用 ZenTao MCP 创建 Bug
   - 若本次带图片路径，必须优先调用支持 `inlineImagePaths` 的创建入口
   - 若当前运行时由 Agent 自行实现描述内贴图，必须先获取创建页 `uid`，先传图，再提交表单
   - 若本次带附件路径，必须优先调用支持 `attachmentPaths` 的创建入口
   - 若创建成功但 `files` 为空，必须立即进入“创建后补传附件”兜底流程
   - 若本次带图片路径，禁止退回普通 `/bugs` REST 创建
   - 只有在确认当前运行时没有该能力时，才允许退回普通创建
8. 获取返回的 Bug ID
9. 立即执行创建后校验

## 创建后校验

创建成功后，不以 `201 Created` 作为唯一依据，必须继续校验。

至少执行以下检查：

1. 查询 Bug 明细
2. 查询产品 Bug 列表，确认能查到该 Bug
3. 如果创建时带了 `project`，再查询项目 Bug 列表，确认能查到该 Bug
4. 如果本次带了图片路径，必须进一步校验 Bug 明细里的 `steps`
   - 只有 `steps` 中出现 `<img` 或 `file-read-` 等可识别的内嵌图片痕迹，才算“图片内嵌成功”
   - 若要求缩小图片，进一步检查 `<img>` 上是否保留了 `max-width:360px;max-height:520px;height:auto;` 或等效缩放样式
   - 若 Bug 已创建但 `steps` 中看不到图片，必须明确告诉用户“Bug 已创建，但描述内贴图失败”
5. 如果本次带了附件路径，必须进一步校验 Bug 明细里的 `files`
   - 只有 `files` 非空，且附件名与本次上传文件名能对应上，才算“附件上传成功”
   - 若创建接口返回成功但 `files` 为空，不得宣称附件已上传成功，必须继续执行“创建后补传”兜底流程
   - 若补传后 `files` 仍为空，必须明确告诉用户“Bug 已创建，但附件自动上传失败”

若产品维度能查到、项目维度查不到，明确告诉用户：

- Bug 已创建成功
- 但项目视图可能不可见
- 当前实际挂载关系是什么

## 返回格式

完成后向用户返回以下内容：

- Bug ID
- Bug 链接
- 标题
- 项目名称与 ID
- 产品名称与 ID
- 是否影子产品
- 指派人姓名与账号
- 当前状态
- 附件上传结果
- 校验结果

若创建失败，优先返回：

- 哪个字段校验失败
- 已尝试的默认补全是什么
- 下一步准备怎么修正

## Bug 链接规则

创建成功后，优先返回可直接打开的 Bug 链接。

链接基于 `ZENTAO_BASE_URL` 拼接，优先使用：

- `<ZENTAO_BASE_URL>/bug-view-<BugID>.html`

若实例兼容参数路由，也可补充返回：

- `<ZENTAO_BASE_URL>/index.php?m=bug&f=view&bugID=<BugID>`

若 `ZENTAO_BASE_URL` 末尾已包含 `/`，拼接时避免重复 `/`。

若当前环境无法拿到 `ZENTAO_BASE_URL`，至少返回 Bug ID，并明确说明未能生成完整链接。

若本次带了附件路径，还必须显式返回：

- 已上传附件名列表
- 使用了哪条附件策略：`创建时上传` 或 `创建后补传`
- 附件校验结果：是否在 `getBugDetail(files)` 中实际查到
- 或明确说明当前环境不支持自动上传附件

若本次带了图片路径，还必须显式返回：

- 已内嵌图片名列表
- 使用了哪条图片策略：`描述内贴图`
- 默认图片样式：`max-width:360px;max-height:520px;height:auto;`
- 图片校验结果：是否在 `getBugDetail(steps)` 中实际看到内嵌图片
- 或明确说明当前环境不支持描述内贴图

## 默认行为

当用户只说“帮我提 bug”“帮我提个禅道 bug”时，默认执行本技能。

此时按以下优先级理解输入：

1. 优先识别显式的 `产品=...`
2. 其次识别显式的 `项目=...`
3. 若未显式标注，先按项目名尝试解析
4. 项目解析失败后，再按产品名尝试解析

## 禁止事项

不要做以下事情：

- 不要直接把项目 ID 当产品 ID 使用
- 不要在未解析指派账号时直接提交中文姓名
- 不要因为缺少模块、版本、类型等非核心字段就立即报错退出
- 不要只报“创建成功”而不做校验
- 不要忽略影子产品关系
- 不要在存在多个候选产品时静默猜测
- 不要在用户提供了附件路径时仍然调用不支持附件上传的普通 `/bugs` 创建入口
- 不要把用户提供的 `图片路径` 直接当成普通附件，除非用户明确要求“作为附件上传”

## 结果说明要求

若最终挂到了影子产品，必须明确写出：

- 项目名
- 项目 ID
- 产品名
- 产品 ID
- `shadow` 情况
- 为什么本次挂到该产品

若最终挂到真实产品，也要明确说明这是最终实际落点。

## 示例输入

### 示例 1：按项目提 Bug

```text
帮我提 bug：
项目=PDF Pro
指派给=陈润国
现象=Android 16，Pixel 6，打开 PDF 后切后台再回来闪退
复现步骤=
1. 打开 PDF Pro
2. 打开任意 PDF 文件
3. 切到后台
4. 再返回应用
期望结果=返回后正常展示 PDF
实际结果=应用闪退退出到桌面
```

### 示例 2：按产品提 Bug

```text
给产品 PDF Reader–File Manager（Launcher）提 bug，指派给陈润国。
现象是打开 PDF 后切后台返回闪退。
复现步骤是……
```
